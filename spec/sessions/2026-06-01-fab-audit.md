## Audit Observations

### #1 — Keyboard Shortcut Does Not Open Menu

**Status:** `validated`

**Root cause:** `chrome.tabs.sendMessage` from the MV3 service worker is blocked on ordinary pages because Reway does not request broad `host_permissions`. The storage-session broadcast replacement was directionally right, but `chrome.storage.session` is trusted-context only by default, so content scripts could not observe the `storage.onChanged` command until the service worker called `chrome.storage.session.setAccessLevel({ accessLevel: "TRUSTED_AND_UNTRUSTED_CONTEXTS" })`. After user validation still failed, the strongest remaining cause is Windows/OS-level interception of `Alt+Shift` before Chrome or page JavaScript receives it.

**Secondary code concern:** If the command lands while `boot()` is still awaiting settings/CSS, the storage listener can fire before `refs.fab` exists.

**Fix:** Service worker now prepares quick-access command storage at startup and before command publish, exposes `storage.session` to content scripts when supported, falls back to `storage.local` otherwise, and writes `{ action: "open-quick-access", ts }` under `rewayAccessCommand`. Content script watches `session`/`local`, opens only when `document.hasFocus()` is true, and queues one command if refs are not ready yet. Default command changed to `Ctrl+Shift+Y` to avoid the OS/input-language `Alt+Shift` path; content script still accepts legacy `Alt+Shift+R` as a best-effort page-level fallback.

**Files:** `extension/background.js`, `extension/access-content-script.js`, `extension/manifest.json`, `spec/fab-access/README.md`

### #2 — Typing Does Not Route to Search Field

**Status:** `validated`

**Root cause:** The only `keydown` listener is on `refs.search` (the search input element itself). In hover-open mode, `focusSearch: false`, so the search input is never focused; keystrokes go to the host page. In keyboard-shortcut mode, `refs.search.focus()` is called, but if the Shadow DOM focus is not correctly acquired (or shortcut is not working per #1), keystrokes still bypass the input.

**Fix:** Added global capture-phase key routing while the quick-access menu is open. Printable keys, Backspace/Delete, and navigation keys now focus `refs.search` and route through the same search/navigation handlers even when Shadow DOM focus is lost or the menu was opened by hover. Normal page typing is untouched while the menu is closed. Search transport now falls back to locally filtering `groupId=all` bookmark data when `/api/extension/bookmarks/search` returns 404, covering deployed hosts that do not yet have the new search route.

**Files:** `extension/access-content-script.js`, `extension/background.js`, `extension/js/api.js`

### #3 — Chevron Arrow on Group Rows

**Files:** `extension/access-content-script.js` — `renderPanel()`, `extension/access-content.css`

### #4 — Scroll Broken in Groups List

**Status:** `validated`

**Root cause:** `activateGroup()` calls `renderPanel()`, which calls `refs.panel.replaceChildren(list)`. This rebuilds the entire DOM on every group hover, destroying and recreating the scroll container and resetting its scroll position. Scrolling while the mouse passes over group rows continuously triggers full re-renders.

**Fix:** Replaced `renderPanel()` inside `activateGroup()` with `updateActiveRows()` so hover/focus only mutates active attributes and keeps the scroll container alive. Group-list scroll now closes the submenu instead of re-anchoring stale hover content while the list moves.

**Files:** `extension/access-content-script.js` — `activateGroup()`

### #5 — Open-Group Button Isolated from Group Surface

**Status:** `validated`

**Root cause:** The `.group-open-button` is a flex sibling of `.row-button` inside `.group-row`. It renders outside the group row's background/border surface with its own visual bubble.

**Fix:** Added a neutral `.group-row-surface` wrapper inside each `<li role="none">`, with the group-select button and open-all button as sibling buttons. The wrapper owns the active/hover/focus-within background, keeping one visual row surface while avoiding button-in-button markup. Added inset row padding and derived inner radii so the open-all button is optically aligned inside the row surface.

**Files:** `extension/access-content-script.js` — `renderPanel()`, `extension/access-content.css`

### #6 — Groups Show Color Dot Instead of Assigned Dashboard Icon

**Status:** `validated`

**Root cause:** `normalizeGroups()` mapped only `id`, `name`, and `color`, discarding the `icon` field already returned by `EXTENSION_GROUP_SELECT`. The group row and submenu header both rendered a CSS color marker instead of the assigned dashboard icon.

**Fix:** The FAB now preserves `group.icon`, loads a generated Hugeicons manifest before the content script, and renders icon chips instead of plain color dots. The chip keeps the group color as a tinted surface while the inline SVG carries the icon itself. Unknown or missing icon keys fall back to the generated default icon, then to a small color dot only if the manifest is unavailable.

**Files:** `extension/access-content-script.js`, `extension/access-content.css`, `extension/manifest.json`, `extension/js/group-icon-manifest.js`, `scripts/generate-group-icon-manifest.mjs`, `package.json`

### #7 — Bookmarks Fall Back to Letter Favicon

**Status:** `validated`

**Root cause:** The FAB trusted only the first stored favicon URL. Any bookmark with `favicon_url = null`, an invalid URL, or a broken favicon request immediately dropped to the letter tile even when a public-domain fallback could still succeed.

**Fix:** Bookmark normalization now builds a validated favicon candidate list. The renderer tries the stored favicon first, then Google S2 for public domains only, then replaces the image with a deterministic letter tile if every candidate fails. Localhost, private-network, and non-public hosts skip the Google fallback entirely.

**Files:** `extension/access-content-script.js`, `extension/access-content.css`

### #10 — Shadows Don't Match Reway Design Language

**Status:** `open`

**Root cause:** `access-content.css` applies `box-shadow` to `.fab`, `.menu`, `.submenu`, `.intro`, `.marker`. Reway's design language (popup, dashboard) does not use decorative drop shadows.

**Fix:** Remove `box-shadow` drop-shadow values from `.fab`, `.menu`, `.submenu`, `.intro`. Retain `inset` ring shadows (border substitutes) — those are structural, not decorative.

**Files:** `extension/access-content.css`

### #11 — Grab Cursor on FAB Hover Is Annoying

**Status:** `open`

**Root cause:** `.fab { cursor: grab; }` applies the grab cursor on hover, even before the user initiates a drag. `.fab[data-dragging="true"] { cursor: grabbing; }` applies during drag.

**Fix:** Change `.fab { cursor: default; }` (or `pointer`). Keep `cursor: grabbing` only for the active drag state via `data-dragging="true"`.

**Files:** `extension/access-content.css`

### #13 — FAB Flashes Tooltip and Empty Menu on Page Load

**Status:** `validated`

**Root cause:** The remaining visible flash was styled, not raw. After the host reveal path completes, `boot()` immediately calls `maybeShowIntro()`, which opens the one-time intro tooltip on page load. That intentional first-run tooltip reads as a flash because it appears before any user interaction. The earlier host-hide change only reduced the unstyled-risk window; it did not address the deliberate intro reveal.

**Fix:** `boot()` no longer auto-calls `maybeShowIntro()` after reveal. The FAB now appears quietly on page load instead of opening the onboarding tooltip by itself. Existing interaction paths still call `dismissIntro()`, so first interaction marks the intro as seen and prevents latent repeat behavior.

**Files:** `extension/access-content-script.js` — `boot()`, `inject()`

## Follow-up Chunks

### X Auto-Capture Toggle

**Status:** `validated`

**Root cause / need:** X bookmark auto-capture was always on whenever the extension was installed. That behavior belongs to extension setup, not the FAB itself, and needed a browser-local opt-out without affecting the existing `X Bookmarks` group or previously saved items.

**Fix:** Added a local popup setting `Capture X bookmarks automatically`, stored in `chrome.storage.local` as `rewayXAutoCaptureEnabled` with default `true`. The X/Twitter content script now bails before sending `twitterBookmark` messages when disabled, and the background worker also re-checks the same key and ignores any stale capture message as a safety net. Turning it off stops new auto-captures only; it does not hide or delete `X Bookmarks`.

**Files:** `extension/popup.html`, `extension/popup.js`, `extension/js/ui.js`, `extension/twitter-content-script.js`, `extension/background.js`
